# Private RAG Apps — 要件定義書 (requirements.md)

> v0.5。アーキテクチャの正は `architecture.md`、データモデルの正は `db_design.md`（本書は概要のみ記載）。

---

## 1. 概要

Private RAG Apps は、**ローカルのプライベートドキュメントコーパス**を取り込み、**出典（引用）付きで回答する RAG チャットアプリケーション**です。

### このプロジェクトの位置づけ

公開リポジトリによる**技術力のショーケース**。「動く RAG」を作ることが目的ではなく、プロダクション GenAI で本当に難しい以下の 4 点に向き合った設計・実装であることを示すことを主目的とします。

1. **検索・回答品質の評価（Eval）** — なんとなく動く、で終わっていない証拠
2. **可観測性（Observability）** — トレース / コスト / レイテンシの計測
3. **クリーンなアーキテクチャ境界** — ingestion / retrieval / generation の疎結合
4. **信頼性** — 出典必須・ハルシネーション抑制

> 設計判断は「なぜそうしたか」をドキュメント・コミット・PR に残すことを重視します。レビュアー（採用側・技術者）が見るのは成果物そのものより、判断の痕跡です。

### v0.2 でのスコープ再定義

SaaS コネクタ（Notion / Slack / Google Drive）と OAuth を **v1 スコープから外し、将来拡張に移動**しました（§11）。理由:

- 単一ユーザー MVP に OAuth フロー（アプリ登録・コールバック・トークン更新）は過剰
- レビュアーが自分の SaaS アカウントなしで**すぐ動かせる**ことを優先（→ FR-7 デモモード）
- RAG の技術的核心（ハイブリッド検索・リランク・Eval・可観測性）はローカルコーパスで完全に示せる

## 2. スコープ

### In scope (v1)

- **単一ユーザー**向け
- **ローカルドキュメントコーパスの取り込み**（Markdown / プレーンテキスト）
- **シードデータ同梱のデモモード**（`make demo` で即チャット可能）
- **増分再取り込み**（コンテンツハッシュによる無変更スキップ、削除反映）
- **ハイブリッド検索**（ベクトル + 全文 / pg_bigm）+ **リランク**
- **出典付きストリーミングチャット**（ストリーミングは M2 で充足 → §10）
- 会話履歴
- 検索・生成品質の **Eval ハーネス**（最小 Eval を M0 から運用 → §9）
- 可観測性（トレース / コスト / レイテンシ。計装は M0 から → §10）

### Out of scope (v1) — 将来拡張として §11 に明示

- **SaaS コネクタ**（Notion / Slack / Google Drive）と **OAuth 認証**
- マルチユーザー / 組織テナント
- **ドキュメント単位の ACL（権限考慮検索）**
- **エージェンティック RAG**（多段検索・クエリ分解・自己評価ループ）
- PDF / Office ファイルのパース（取り込みは Markdown / テキストに限定）
- モバイルアプリ

## 3. 用語

| 用語 | 定義 |
|---|---|
| コーパス (Corpus) | 取り込み対象のドキュメント集合。v1 ではローカルディレクトリ配下の Markdown / テキスト。 |
| シードデータ | リポジトリに同梱するサンプルコーパス。デモモードと Eval ゴールデンデータセットの土台を兼ねる。 |
| チャンク (Chunk) | ドキュメントを検索単位に分割したテキスト。埋め込みと全文インデックスの対象。 |
| ハイブリッド検索 | ベクトル検索（意味的類似）と全文検索（語彙一致）を融合した検索。 |
| RRF | Reciprocal Rank Fusion。複数の検索結果を順位ベースで融合する手法。 |
| リランク (Rerank) | 一次検索で得た候補を、クロスエンコーダ等でクエリとの関連度に応じて並べ替える工程。 |
| Faithfulness | 生成回答が、取得コンテキストに忠実か（コンテキスト外の主張をしていないか）の指標。 |
| TTFT | Time To First Token。ストリーミングで最初のトークンが返るまでの時間。 |
| 棄権 (Abstain) | コンテキストが弱い/無関係な場合に、生成が推測で答えず「該当する情報が見つかりませんでした」を返すこと（FR-4）。 |
| Condense | マルチターン会話のフォローアップ質問を、直前の会話履歴を踏まえて自己完結したクエリに書き換える処理（FR-5）。 |

## 4. 機能要件

### FR-1 コーパス管理

- 取り込み対象ディレクトリを設定し、配下の **Markdown (.md) / プレーンテキスト (.txt)** を取り込み対象とする
- 対応形式外のファイルはスキップし、スキップ理由をログに残す
- 取り込み済みソースの一覧（ファイルパス・タイトル・取り込み日時・チャンク数）を確認できる
- コーパス全体の削除（インデックス初期化）ができる

### FR-2 データ取り込み (Ingestion)

- ドキュメントを共通の内部表現に正規化する
- **チャンキング**: 見出し境界を尊重した分割（オーバーラップ付き）
- チャンクごとに**埋め込み**を生成し、ベクトル + 全文インデックスとして保存（upsert）
- **増分再取り込み**: `content_hash` を比較し、無変更ドキュメントの再埋め込みをスキップする（コスト削減）
- ソース側で削除されたドキュメントはインデックスからも除外する（論理削除）
- 取り込みは CLI（`make ingest`）から実行できる

### FR-3 検索 (Retrieval)

- クエリに対し **ベクトル検索** と **全文検索（pg_bigm）** を並列実行し、**RRF で融合**する
- 融合結果の上位を **リランク** し、最終的な top-k コンテキストを決定する
- 各チャンクに出典メタデータ（タイトル / パス / 見出しパス / 更新日時）を保持する

### FR-4 生成 (Generation)

- 取得したコンテキストに**基づいてのみ**回答を生成する
- 回答には**必ず出典を付与**する
- コンテキストに答えがない場合は、推測せず「該当する情報が見つかりませんでした」と返す
  - 注: 検索は関連の弱いチャンクでも top-k 件返すのが通常のため、この要件の実体は「**弱い/無関係なコンテキストで生成が棄権（abstain）できること**」である（検索 0 件時の短絡は例外パス）。Eval の negative ケース（§9）はこの棄権を検証する
- 回答はストリーミングで返す（M2 で充足。M0〜M1 は非ストリームでよい → §10）

### FR-5 チャット体験

- マルチターン会話。フォローアップ質問では**会話履歴を考慮したクエリ書き換え**を行う（例:「それの詳細は?」→ 直前の話題を補完）
- 回答本文と**出典カード**（クリックで元ソース情報を表示）を表示する
- ストリーミング表示（TTFT を短く）

### FR-6 会話履歴

- 会話の保存・一覧・再開ができる

### FR-7 デモモード ★レビュアー体験の核

- **シードコーパスをリポジトリに同梱**し、`make demo` の 1 コマンドで「シード投入 → 取り込み → チャット可能」まで到達できる
- シードコーパスは日本語を含む現実的な文書構成とし、**Eval のゴールデンデータセット（§9）と兼用**する
- README のクイックスタートはデモモードを前提に書く

### FR-8 データ管理 UI

- 取り込み済みソース一覧・チャンク数・最終取り込み日時の表示
- 再取り込みトリガ・インデックス初期化

## 5. 非機能要件

### NFR-1 検索・回答品質 ★ショーケースの核

ゴールデンデータセットに対して以下を計測し、**ベースライン → 改善目標**を追跡する。

| レイヤ | 指標 | 規模・条件 |
|---|---|---|
| Retrieval | Recall@5, Recall@10, nDCG@10, MRR | ゴールデンデータセット 30〜50 問 |
| Generation | Faithfulness, Answer Relevance | LLM-as-judge（判定モデルは GPT 系の軽量モデルに固定し、モデル名を記録） |

- プロンプト・検索ロジック・チャンキング変更時に**回帰を検出**できること（→ §9, CI 連携）
- 最小 Eval（10 問 + Recall@5）は **M0 から**運用する（→ §10）
- 正解ラベルは **文書（path）レベル**で持つ（chunk_id 非依存。チャンキング変更後もデータセットを再利用するため → `docs/specs/m3_eval_expansion.md` §3.3）
- Recall@10 計測のため、評価時は本番 `top_k` と別の **`EVAL_TOP_K`（≥10）** で取得する
- CI ゲートは **検索指標=ハード / 生成指標=ソフト**（LLM-as-judge のノイズを許容。同 §7.3）

### NFR-2 パフォーマンス

- TTFT（初回トークンまで）の目標値を設定（例: p95 < 2.0s）※ M2 で実測しベースラインから暫定確定（→ `docs/specs/m2_streaming_and_history.md` §7）
- 検索レイテンシ p95 目標（例: < 500ms、リランク込み）※ 同上
- 計測は**初回ターン / フォローアップを分けて**行う（フォローアップは condense の LLM 呼び出しが TTFT に直列に乗るため、同一目標にしない）

### NFR-3 セキュリティ / プライバシー

- シークレット（API キー等）は環境変数 / `.env` 管理。**リポジトリへのコミット禁止**
- **ローカルファースト**: コーパスとインデックスは自分のインフラ（ローカル PostgreSQL）に閉じる
- 実データ（個人のプライベート文書）をリポジトリ・テスト・Eval データに含めない（同梱するのはシードデータのみ）
- データ削除: インデックス初期化・全消去の導線を用意する

### NFR-4 可観測性 ★ショーケースの核

- すべての **LLM / 埋め込み / リランク呼び出しをトレース**（Langfuse）
- トークン数・コストを呼び出し単位で記録し集計する
- 主要処理のレイテンシをスパンとして計測する
- 計装は **M0 の骨格段階で配線**する（後付けはトレース漏れを生むため → §10）
- `LANGFUSE_*` 未設定時は計装を **no-op（graceful degradation）** とし、アプリ・デモ・Eval の動作を妨げない（NFR-8 のレビュアー体験を優先）

### NFR-5 コスト管理

- 呼び出しごとのコスト記録と集計は **Langfuse の標準機能で可視化**する（自作ダッシュボードは v1 では作らない）

### NFR-6 信頼性 / ハルシネーション対策

- 出典に無い主張はしない（Faithfulness を Eval で監視）
- 低信頼時は「見つからない」を返す

### NFR-7 保守性

- ingestion / retrieval / generation の疎結合（→ AGENTS.md の依存方向ルール）
- 型付け（mypy）・テスト・CI を整備

### NFR-8 再現性 ★レビュアー体験の核

- クリーンな環境で `git clone` → `docker compose up` → `make demo` により **15 分以内にチャット可能**な状態へ到達できる
- 必要な外部アカウントは **OpenAI / Voyage の API キーのみ**（SaaS アカウント不要）。**Langfuse は任意**（未設定でもデモは完動し、設定すればトレースが有効になる → NFR-4）

## 6. アーキテクチャ概要

構成の正は `architecture.md` を参照。要点のみ:

- **Ingestion Path**（コーパス → 正規化 → チャンク → 埋め込み → upsert）と **Query Path**（クエリ書換 → ハイブリッド検索 → RRF → リランク → 引用付き生成）を分離する
- ストアは PostgreSQL + pgvector に集約（ベクトル + 全文 + リレーショナル）
- LLM 呼び出しは generation 層、埋め込みは ingestion / retrieval 層に閉じる（→ AGENTS.md）
- すべての AI 呼び出しは Langfuse でトレースする

## 7. 技術選定(決定と根拠)

### 7.1 フロントエンド

| 領域 | 採用 | 根拠 |
|---|---|---|
| フレームワーク | Next.js (App Router) | 既存スキル活用、ストリーミング UI、型共有 |
| チャット UI | **assistant-ui**（shadcn/ui ベース） | バックエンド非依存のカスタムランタイムで自前 SSE を直接受けられる。streaming/auto-scroll/retry を再実装不要。ChatKit は OpenAI API 密結合のため不採用 |
| 状態管理 | Zustand | 軽量・ボイラープレート最小。assistant-ui のランタイム状態と分離したアプリ側 UI 状態（コーパス管理・デモモード表示等）を扱うのに十分 |

### 7.2 バックエンド

| 領域 | 採用 | 根拠 |
|---|---|---|
| 言語 | Python 3.13 | GenAI エコシステムが最も厚い |
| パッケージ管理 | uv | 高速・モダン、showcase として印象良 |
| API | FastAPI + uvicorn | async / SSE ストリーミングが素直 |
| ストア | PostgreSQL + pgvector | **1 DB でベクトル + 全文** を扱え、ハイブリッド検索が組みやすい。運用が軽い |
| ORM | SQLAlchemy | pgvector 拡張との親和性、Alembic との連携が標準的 |
| マイグレーション | Alembic | 定番 |
| 全文検索 | **pg_bigm**（bigram） | 標準 FTS は日本語の分かち書き不可。pg_bigm は拡張 1 つで日本語対応・軽量 |
| LLM | OpenAI GPT | 低コスト・低レイテンシ重視で採用。他のLLMにも切り替えられるように作る |
| 埋め込み | Voyage (voyage-4-lite / 1024次元) | 現行世代・多言語対応・最初の2億トークン無料 |
| リランク | Voyage rerank-2.5 | 埋め込みと同ベンダで完結 |
| トレース/コスト | Langfuse | LLM トレース + コスト集計を一元化 |
| 取り込み実行 | CLI（`make ingest` / `make demo`） | ローカルコーパスの MVP にジョブキューは過剰。SaaS コネクタ導入時に ARQ (Redis) を再検討 |

## 8. データモデル(概要)

物理設計（DDL・インデックス・検索 SQL）の正は `db_design.md` を参照。主要エンティティ:

- **Source** — 取り込み元ドキュメント（パス・タイトル・content_hash・deleted_at）
- **Chunk** — 検索単位（content・embedding・metadata・position）
- **Conversation / Message** — 会話と発話（citations を jsonb で保持）
- **IngestRun** — 取り込み実行ログ（added / updated / deleted / skipped）

## 9. 評価(Eval)計画 ★ショーケースの核

- **ゴールデンデータセット**: シードコーパス（FR-7）に対する質問と、期待される出典（**文書 path レベルの正解ラベル**。chunk_id 非依存 → NFR-1）・参考回答を用意。目標 30〜50 問。コーパスに答えの無い **negative ケース**を必ず含める（FR-4 の棄権検証）
- **指標**: §NFR-1 の Retrieval / Generation 指標
- **段階導入**:
  - **M0: 最小 Eval**（10 問 + Recall@5 のみ）を導入し、以降のすべての検索・プロンプト変更の前後で実行する
  - **M3: 拡充**（30〜50 問、生成指標、CI 連携）
- **実行**: `make eval` でローカル実行。CI では **プロンプト・検索・チャンキング・生成・evals 変更を含む PR** で自動実行し、**committed baseline** からの劣化を検出（検索=ハードゲート / 生成=ソフト。詳細 → `docs/specs/m3_eval_expansion.md` §7）
- 結果はスコアの推移として記録（回帰の可視化）

> 最小 Eval を M0 に置く理由: M1（ハイブリッド化 + リランク）は本プロジェクト最大の検索変更であり、計測なしで実施すると「Eval がショーケースの核」という主張と矛盾するため。

## 10. マイルストーン(Walking Skeleton 方式)

| M | 内容 | 充足する要件 |
|---|---|---|
| **M0 Walking Skeleton** | シードコーパス取り込み(CLI) → ベクトル検索のみ → 非ストリーム回答。**Langfuse 計装の配線**・**最小 Eval(10問/Recall@5)**・**基本 CI(lint/test)** を含む | FR-2(基本), NFR-4, NFR-7(基本CI), §9最小Eval |
| **M1** | ハイブリッド検索(pg_bigm + RRF)+ リランク。**Eval before/after を記録** | FR-3, NFR-1 |
| **M2** | ストリーミング(SSE)+ 出典 UI + 会話履歴 | FR-4(完全), FR-5, FR-6 |
| **M3** | Eval 拡充(30〜50問・生成指標・CI 連携) | NFR-1(完全), §9 |
| **M4** | 増分再取り込み・データ管理 UI・デモモード仕上げ | FR-1, FR-7, FR-8, NFR-8 |
| **M5** | 仕上げ(README・デモ GIF・アーキテクチャ図・ドキュメント整備) | Definition of Success |

> まず M0 で end-to-end を貫通させ、以降は各レイヤを厚くしていく方針。
> FR-4 の「ストリーミングで返す」は M2 で充足する(M0〜M1 の非ストリームは経過状態であり要件違反ではない)。
