## 1. 目的と背景

M0〜M4 で実装は縦横に完成した。M5 は**それを公開リポジトリのショーケースとして"読める"状態に仕上げる**フェーズ。requirements §1 の核心はここに効く:

> 「動く RAG」を作ることが目的ではなく、プロダクション GenAI で難しい 4 点（**Eval / 可観測性 / クリーンな境界 / 信頼性**）に向き合った設計・実装であることを示す。**レビュアー（採用側・技術者）が見るのは成果物そのものより、判断の痕跡**。

したがって M5 の成否は「新しいコードを書いたか」ではなく、**レビュアーが短時間で (a) 動かせて (b) 品質が計測されていると分かり (c) なぜその設計かを辿れる** かで決まる。M5 は次を仕上げる:

1. **README**（フロントドア: 位置づけ・クイックスタート・デモ GIF・アーキ図・スタックと根拠）
2. **Eval レポート**（`docs/eval_report.md`。★ショーケースの核。ベースライン→改善のスコア推移）
3. **可観測性の提示**（Langfuse トレースのスクショ + 説明）
4. **設計判断の可視化**（判断の痕跡＝ADR 索引。§1 に最も効く）
5. **設計文書と実装の一致監査**（requirements/architecture/db_design/AGENTS/specs vs 実コード）
6. **リポジトリ整備 + 15 分再現性の最終確認**（LICENSE・秘匿/実データ最終チェック・README 手順の実走）

> M5 は**新機能を追加しない**。既存の成果を素材化し、整合を監査するフェーズ（AGENTS.md §11 のスコープ規律を最後まで守る）。

## 5. Eval レポート `docs/eval_report.md`（★ショーケースの核）

### 5.1 位置づけと M3↔M5 境界

- **M3 が所有**: ゴールデンデータセット、`make eval`、committed baseline、**自動生成される機械可読 JSON + 人間可読 Markdown サマリ**（M3 §6.2）。数値・メトリクス表・provenance は M3 の生成物。
- **M5 が所有**: レビュアー向けの**物語（narrative）**と**解釈**。「なぜハイブリッドか」「リランクが何を稼いだか」を、M3 の生成数値を**引用**して読ませる。

### 5.2 レポートの骨子（物語 = ベースライン → 改善）

1. **狙い**: 何を測っているか（Retrieval: Recall@5/10・nDCG@10・MRR / Generation: Faithfulness・Answer Relevance。NFR-1）。
2. **データセット**: 規模（30〜50 問）・種別（lookup/synthesis/**negative**）・**path レベル正解**（M3 §3.3）。
3. **スコア推移（中核）**: **M0 ベクトルのみ → M1 ハイブリッド → M1 +リランク**の before/after 表。加えて M3 の**リランク前/後の診断**でリランク寄与を示す（M3 §4.2）。
4. **生成品質**: Faithfulness / Answer Relevance と、**negative ケースの棄権成功率**（信頼性ピラー、NFR-6）。
5. **provenance**: 使用モデル名・decoding 設定・検索パラメータ・corpus ハッシュ・dataset version（M3 §6.2）。
6. **限界と今後**: judge のノイズ・データセット規模・マルチターン評価が sanity 止まりであること（誠実さを示す）。

### 5.3 ★数値の誠実性（fabrication を作らない設計）

- **手書きの数値を置かない**。スコア表・メトリクスは **M3 の `make eval` 生成物（人間可読 Markdown サマリ。M3 §6.2）を一次ソース**とし、**再実行で同じ数字が出る**ことを担保（provenance 付き）。
- **★方式（決定）: 新規の差し込み機構を作らず、M3 サマリを引用する**。`docs/eval_report.md` は「**手書きの解釈テキスト**」＋「**M3 生成サマリへの引用/リンク（該当表の抜粋）**」で構成する。数値を M5 側のテンプレートに機械差し込みする仕組みは**新設しない**（＝「新機能を追加しない」M5 スコープ §2.2 と両立）。数値の最新化は M3 サマリの再生成で行い、eval_report.md 側は解釈だけを更新する。
  - ※ もし引用ではなく自動埋め込みを望むなら、それは **M3 のレポート生成機能の拡張**として M3 スペックに追記すべき事項（M5 では持たない）。M3 §6.2 の Markdown サマリが「eval_report から引用できる粒度・見出し構造」であることを M5 の前提として確認する（§14 反映）。
- レポートは数値（M3 由来・再現可能）と解釈（手書き）を明確に分離。**チェリーピック・捏造は 1 数値でも避ける**（NFR-1 の信頼性とショーケースの誠実さの核）。

## 6. 可観測性の提示（Langfuse トレース）

- **提示物**: 代表的なトレースのスクリーンショット + 説明（README か `docs/observability.md`）。
  - **chat トレース**: 1 リクエスト = 1 トレースで `condense → embed_query → retrieve(vector/fts) → rerank → generate` の span、各のトークン/コスト/レイテンシ、`ttft_ms`（M2 §6）。

## 8. 設計判断の可視化（`docs/decisions.md`）★§1 に最も効く

§1 が「レビュアーが見るのは判断の痕跡」と言う以上、**散在する設計判断を 1 索引に集約**するのが M5 で最も費用対効果が高い。

- **形式**: ADR 的な短い項目の索引（決定・背景・代替案・根拠）。**新規に考えるのではなく、既存 specs / requirements §7・§11 / architecture §11 の判断を集約・リンク**する。
- **載せる代表的判断**（例）:
  - pgvector 単一 DB（vs Qdrant）／pg_bigm（vs PGroonga）／RRF 融合／リランクを最終段（requirements §7、architecture §11）
  - ジョブキューを持たない（CLI + BackgroundTasks）／SaaS・OAuth をスコープ外（requirements §11）
  - 会話の **done 一括保存**（M2 §4.2）／citations 生成前送出（M2 §4.5）
  - Eval の **path レベル正解**・`EVAL_TOP_K` 分離・検索ハード/生成ソフトのゲート（M3）
  - 増分の **埋め込み事前・短トランザクション全置換**・**削除安全弁**・**advisory lock 排他**（M4）
- **重複を作らない**: 各項目は詳細を持たず、正（specs/設計文書）へリンクする索引に徹する。**`decisions.md` は「なぜ（判断・代替案・根拠）」に特化し、「何を（スタック一覧・スコープ定義）」は requirements §7/§11 へリンク**する（二重管理を避ける）。

## 9. 設計文書 ⇄ 実装の一致監査

Definition of Success の「設計文書一式が実装と一致」を**能動的に監査**する（前倒しで文書を更新してきたぶん、実装との差分をここで確定させる）。

- **★是正の方向性ルール（重要）**: 本プロジェクトは設計文書を**実装に先行して**更新してきた（requirements v0.4 等）。そのため監査は「文書に合わせて実装を直す」方向に働きやすいが、それは**禁止**する。原則は:
  - 実装が spec より妥当な判断をしていた → **spec を実装に合わせて更新**し、逸脱理由を残す（AGENTS §12）。
  - 文書の記述が実装と食い違う（文書側の想定が古い/誤り）→ **文書を実装の現実に合わせる**。
  - **文書都合で動くコードを歪めない**。M5 は新機能・仕様変更を伴わない（§2.2）。挙動を変える是正が必要になったら、それは M5 ではなく該当マイルストーンの追加作業として切り出す。
  - 監査は「文書 ⇄ 実装」の**双方向**で、正は原則**実装（動いている現実）**側に置く。

- **API**: architecture §7 のエンドポイント表 ⇄ 実ルート（M2 の `POST /api/conversations`・SSE、M4 の sources/ingest/index が揃っているか）。
- **DB**: db_design §4/§5 の DDL・インデックス ⇄ 実マイグレーション（0001_init）。**「DDL 変更なし」を貫けたか**（M2〜M4 は advisory lock 等で回避）を確認。
- **依存方向**: AGENTS §3 のルール ⇄ 実 import（LLM は generation/evals のみ、埋め込みは ingestion/retrieval のみ 等）。可能なら import-linter 等で機械チェック。
- **specs**: `docs/specs/mN_*` が**実装された姿**と一致（実装中に妥当な逸脱があれば、AGENTS §12 に従い spec を後追い更新して痕跡を残す）。
- **設定キー**: 各 spec で定義した `*_*` 設定 ⇄ `core/config.py` / `.env.example`。
- ドリフトが見つかったら**是正コミット**（`docs:` or コード側）を作り、監査結果を PR に残す。
