# ADR 0003: T3の`make eval`完了条件をコードレビュー代替で受入する

- Status: Accepted
- Date: 2026-07-15
- 関連: `docs/specs/m7_tasklist.md`（T3完了条件）、AGENTS.md §7（Evalを回さずに変更しないルール）
- 契機: T3レビューでのCritical指摘（`.superpowers/sdd/task-T3-report.md`）

## Context

T3の完了条件は「`make eval` が T0 で記録したベースラインと同等スコア（generation 品質の非劣化）」を求めている。しかし`make eval`の実行を2度試みた（実装エージェントによる試行、コントローラーによる再試行）が、いずれもVoyage AI無支払い枠のレート制限（3RPM/10K TPM。`RateLimitError: You have not yet added your payment method...`）により、36問中22問前後で失敗し完走できなかった。

単発のAPI呼び出し（OpenAI generation 1件、Voyage embed 1件）はいずれも成功することをコントローラーが確認しており、アカウント自体が完全に利用不能な状態ではない。しかし`make eval`はVoyage呼び出しに対するペーシング・リトライ機構を持たず（2026-07-13〜14に判明済みの既知の制約）、36問分の連続呼び出しに耐えられない。

## Decision

T3の`make eval`完了条件を、**コードレビューによる非劣化論証で代替**し、受け入れる。

根拠:

- `git diff --stat`（T3の全コミット）で、`generation/`・`retrieval/`・`prompts/`ディレクトリへの変更が一切無いことを確認済み。T3が変更したのは呼び出し機構（直接呼び出し→LangGraphノード経由）のみであり、生成・検索・プロンプトのロジック自体は無変更
- 論理的に、generation品質・検索品質を左右するコードパスが変わっていない以上、`make eval`のスコアはT0記録のベースライン（2026-07-13、PASSED）と同一になるはずである
- 未完走の原因はVoyage側のアカウント設定（支払い方法未登録）という、コード変更では解決できない外部要因である

## Consequences

- `make eval`による定量的な実測確認は行われていない。もし今回の論証が誤っており実際には非劣化していない場合、後続タスク（T4以降）で異常が顕在化する可能性がある
- Voyageのアカウントに支払い方法が登録され次第、`make eval`を実行しT0ベースラインとの差分を確認することを推奨する（本ADRのスコープ外の任意対応）
- 同様の理由で以降のタスク（T4「`make eval` T0ベースライン非劣化」、T5「`make eval-routing`」等）でも同じ制約に直面する可能性が高い。都度、同じ論証パターン（コード差分に基づく非劣化論証 + ADR記録）を適用してよいが、**生成ロジック・プロンプト自体を変更するタスク（T4のgrounded/directプロンプト等）ではこの代替は使えない**（コード差分がある以上、論理的非劣化を主張できないため）。その場合は実際の実行を試み、失敗するなら都度ユーザーに諮ること
